01 / OVERVIEW
§ 01 — The Two Approaches

Same Goal, Opposite Philosophies

SecureExam Browser KIOSK APP

What it is: a standalone Electron application the student installs and launches. It seizes the whole machine for one exam session.

Where it enforces: the operating system & network layer — below the web page.

Core idea: “No student cooperation required.” It blocks escape routes at the OS itself.

Footprint: Windows (full), Android (partial), PWA (weak). Installer, admin rights, registry edits.

Prodigy Shield EXTENSION

What it is: a Chrome / Brave browser extension that runs inside the exam tab on the Prodigy Review site.

Where it enforces: the browser & web-page layer — inside the tab.

Core idea: “Lightweight, instant, cross-platform.” Detect, flag, and progressively lock out.

Footprint: any desktop Chromium browser — Windows, macOS, Linux, ChromeOS. No install ceremony.

03 / SEB
§ 03 — SecureExam Browser KIOSK

SEB — Where It Wins

Strengths
+

OS-level lockdown. Kiosk fullscreen with no chrome or title bar — the student physically cannot leave the session by ordinary means.

+

~40 global hotkeys intercepted before the OS reacts, plus registry edits that disable Task Manager and hide the taskbar.

+

Screen capture defeated. The window renders solid black in screenshots, Snipping Tool, and screen recorders.

+

Network-layer AI block. 25+ AI domains (and all subdomains) denied; admin-extensible.

Strengths (cont.)
+

Clipboard control + poison. Copy/paste disabled and the clipboard is wiped (and pre-poisoned) on exit.

+

Tamper-resistant audit trail. Append-only JSONL log survives the session for evidence.

+

No student compliance assumed. Enforcement lives below the web page — not dependent on goodwill.

+

Offline-capable. Local salted-SHA-256 auth; no remote dependency for controlled labs.

Highest ceiling of enforcement — when you control the hardware.
04 / SEB
§ 04 — SecureExam Browser KIOSK

SEB — The Trade-offs

Deployment Pain
✕

Very hard on macOS. Needs a separate signed/notarized build; several Windows tricks (registry, taskbar, display-affinity) have no clean Mac equivalent.

✕

Impossible on iOS / iPad. Apple forbids kiosk-style third-party browsers; the whole model can't ship there.

✕

Heavy install. Every student must download an app, grant admin rights, and clear a first-run setup — a support burden at scale.

✕

Per-OS maintenance. Windows, Android, and PWA are three codepaths to build, sign, and keep in sync.

Security Gaps
✕

PWA target is weak — fullscreen and Service-Worker interception are bypassable by opening another browser.

✕

Admin bypass via registry edit is possible on an unmanaged machine.

✕

No lockout / appeal workflow — it records events but doesn't manage the incident.

✕

Doesn't stop out-of-band cheating — a second phone camera or device is untouched.

⚠
The headline issue: SEB is painful-to-impossible to deploy on Mac and iOS. Its strongest defenses are Windows-specific — so mixed-device cohorts fall outside its reach.
05 / PRODIGY
§ 05 — Prodigy Shield EXTENSION

Prodigy — Where It Wins

Strengths
+

Cross-platform by default. Runs on any desktop Chromium browser — Windows, macOS, Linux, ChromeOS — the exact gap SEB can't cover.

+

Near-zero install friction. Load the extension in seconds; no admin rights, no registry, no installer.

+

Preflight check verifies the environment (fullscreen, DevTools, browser, tab) before the exam starts.

+

Progressive lockout with integrity flags, an exit limit, and automatic penalty lockdown.

Strengths (cont.)
+

Built-in appeal + admin review. Locked students can appeal; admins resolve it from a live control center.

+

Remote config. Rules (flag thresholds, exit limits, clipboard policy) update centrally without redeploying.

+

Single codebase. One extension to maintain instead of per-OS native builds.

+

Fast to iterate. Ship fixes the same day — no app-store or installer cycle.

Best reach and lowest rollout cost — ideal for bring-your-own-device cohorts.
06 / PRODIGY
§ 06 — Prodigy Shield EXTENSION

Prodigy — The Trade-offs

Depth Limits
✕

Page-level only. It can't reach the OS — no true screenshot blackout and no global Alt+Tab / Task-Manager block.

✕

Student can disable it. The extension can be toggled off or the browser closed outside the exam's guarded flow.

✕

Softer signals. Tab-switch and focus loss are detected and flagged — not physically prevented.

✕

Depends on a backend. Lockout, config, and appeals need the server reachable.

Coverage Limits
✕

Chromium only. Needs Chrome / Edge / a Chromium browser; Brave is blocked at preflight, and Safari/Firefox aren't supported.

✕

No iOS / iPad. Mobile Safari doesn't allow this class of extension — same wall SEB hits.

✕

No clipboard poisoning or network AI-domain deny-list at SEB's depth.

✕

Out-of-band cheating (second device, phone camera) is unaddressed — as with SEB.

ℹ
Prodigy trades raw depth for reach and speed. It raises the effort to cheat without owning the machine — but a determined student with full device control has more room than under SEB.

08 / VERDICT
§ 08 — Recommendation

When To Use Which

Choose SEB

Controlled Windows labs, high-stakes

When you own the hardware and the exam is high-stakes, SEB's OS-level lockdown is unmatched.

Ideal: on-site Windows lab, proctored certification, managed device pool.
Avoid when: students bring Macs / iPads, or you can't install software.
Choose Prodigy

BYOD, cross-platform, fast rollout

When students use their own varied devices and you need to launch quickly, Prodigy's reach and lockout/appeal flow win.

Ideal: remote / distributed exams, mixed Windows + Mac, quick deployment.
Avoid when: you need screenshot blackout or OS-level hotkey blocking.
◆
Better together: SEB for supervised Windows labs, Prodigy for everyone on Mac / BYOD — and pair either with randomized question pools, time limits, and human supervision. Neither solves iOS or out-of-band cheating alone.
